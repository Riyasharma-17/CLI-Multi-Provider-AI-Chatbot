# Day 6 & Day 7 — Refactoring and Multi-Provider Architecture

---

# 🚀 The Project Started Growing

By Day 5, the chatbot was working. It could:

* Talk continuously
* Remember previous messages
* Handle system prompts
* Support JSON mode

But one problem became obvious.

`main.py` had become huge.

Everything lived in one file:

* API code
* JSON handling
* Commands
* System prompts
* Provider selection
* Memory

It worked, but it was becoming messy.

This is exactly the point where real software engineering starts.

---

# 🧹 Refactoring Begins

Instead of adding more features to one giant file, we decided to separate responsibilities.

We learned an important idea:

> A file should have one responsibility.

---

# Provider Layer

We created:

```text
providers/
```

and moved API-specific code there.

## groq_provider.py

Groq API key loading, client creation and response extraction were moved here.

Instead of:

```python
client.chat.completions.create(...)
```

inside `main.py`, we created:

```python
chat_with_groq(messages)
```

which returns:

```python
bot_reply
input_tokens
output_tokens
```

Now `main.py` doesn't know anything about Groq internals.

---

# Why?

Suppose Groq changes:

```python
response.choices[0].message.content
```

to:

```python
response.answer.text
```

Only one file would need modification.

This is called **Abstraction**.

---

# JSON Utilities

JSON mode had become ugly.

We had:

* json.loads()
* json.dumps()
* try
* except
* pretty printing

inside main.py.

So we created:

```text
utils/json_utils.py
```

and moved everything there.

Now main.py simply says:

```python
display_response(bot_reply, json_mode)
```

without caring how JSON parsing works.

---

# System Prompt Utilities

Earlier:

```python
system_message = {
    "role":"system",
    "content":"You are helpful."
}
```

was hardcoded.

We wanted personalities to be customizable.

So we built:

```text
utils/system_utils.py
```

Now users can enter:

```
You are a Python mentor.
```

or

```
Explain like I'm five.
```

without touching code.

---

# Command Utilities

Soon commands started increasing.

We already had:

* /quit
* /clear

and more commands would come later.

Instead of dozens of if statements in main.py, we created:

```text
utils/command_utils.py
```

which handles commands separately.

This made main.py cleaner and future-proof.

---

# JSON Mode Utility

Earlier:

```python
if user_input.startswith("/json")
```

was inside the main loop.

We moved that into:

```text
utils/json_mode_utils.py
```

which returns:

```python
json_mode, user_input
```

Main.py no longer worries about parsing commands.

---

# Provider Selection Utility

Provider selection was also moved into:

```text
utils/provider_utils.py
```

The chatbot could now choose between providers without cluttering main.py.

---

# 🏗 Architecture Started Emerging

Our project now looked like:

```text
main.py

providers/
    groq_provider.py

utils/
    json_utils.py
    command_utils.py
    system_utils.py
    json_mode_utils.py
    provider_utils.py
```

For the first time, the project looked like a real software project instead of a single script.

---

# Debugging Lessons

One day the chatbot suddenly crashed.

It showed:

```
Using: None
```

followed by:

```
NameError: bot_reply is not defined
```

The problem wasn't in the chatbot.

The real bug was inside:

```python
choose_provider()
```

We had written:

```python
provider = "groq"
```

instead of:

```python
return "groq"
```

Python returned None automatically.

Because provider became None:

* Neither Groq nor Gemini code executed.
* bot_reply was never created.
* NameError occurred later.

This taught us an important lesson:

> Sometimes the error appears far away from the real bug.

---

# Bringing Gemini Back

We wanted true multi-provider support.

So we created:

```text
providers/gemini_provider.py
```

Gemini required a different approach.

Unlike Groq, it doesn't naturally accept message lists.

Therefore we converted:

```python
messages
```

into a single conversation transcript:

```python
user: ...
assistant: ...
```

and sent that to Gemini.

This simulated memory.

---

# Real World Problems

Gemini kept failing.

First:

```
404 model not found
```

Then:

```
429 quota exceeded
```

Even after creating a new API key, Google still gave:

```
limit : 0
```

The code was correct.

The infrastructure wasn't.

This taught another lesson:

> Not every error is a coding mistake.

Sometimes external services fail.

---

# Error Handling

Instead of crashing the entire program, we added:

```python
try:
    ...
except Exception as e:
```

Now provider failures return:

```python
Gemini Error: ...
```

while keeping the chatbot alive.

This made the system more robust.

---

# OpenRouter Integration

Since Gemini quotas were problematic, we added another provider:

```text
providers/openrouter_provider.py
```

The same interface was reused:

```python
bot_reply
input_tokens
output_tokens
```

This proved that our abstraction was successful.

Main.py didn't care whether the answer came from:

* Groq
* Gemini
* OpenRouter

Every provider behaved the same.

---

# Another Real-World Bug

We tried:

```
deepseek/deepseek-chat-v3-0324:free
```

and OpenRouter returned:

```
404 model unavailable for free
```

Eventually we discovered:

```python
MODEL_NAME = "openrouter/free"
```

which worked perfectly.

This taught us:

> APIs change. Good architecture survives those changes.
